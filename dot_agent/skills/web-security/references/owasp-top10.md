## A01:2021 - Broken Access Control

### 概要
アクセス制御の不備により、ユーザーが自分の権限を超えた操作を実行できてしまう脆弱性。

### リスク
- 未認証・未認可のデータアクセス
- 権限昇格（垂直・水平）
- 機密情報の漏洩
- データの不正な変更・削除

### 主な攻撃例
- URL の直接操作による他ユーザーのリソースへのアクセス
- ID パラメータの改ざん（IDOR: Insecure Direct Object Reference）
- API エンドポイントへの無認可アクセス
- メタデータ操作による権限昇格

### 対策

#### 基本原則
- **デフォルト拒否**: すべてのリソースはデフォルトで拒否し、明示的に許可する
- **最小権限の原則**: 必要最小限の権限のみを付与
- **サーバーサイドで検証**: クライアント側の制御に依存しない

#### 実装レベル
1. **認証チェック**
   - すべてのリクエストでユーザーの認証状態を確認
   - セッションの有効性を毎回検証

2. **認可チェック**
   - リソースごとに所有者やアクセス権限を検証
   - 水平権限昇格の防止: ユーザーID と対象リソースの紐付けを確認
   - 垂直権限昇格の防止: ロールベースの権限チェック

3. **リソースの保護**
   - ディレクトリリスティングの無効化
   - メタデータファイル (.git, .env など) へのアクセス禁止
   - バックアップファイルの公開防止

4. **API の保護**
   - RESTful API の全エンドポイントに認可チェックを実装
   - レート制限の導入
   - CORS の適切な設定

5. **ログとモニタリング**
   - アクセス制御の失敗をログに記録
   - 連続した失敗を検知してアラート

## A02:2021 - Cryptographic Failures

### 概要
暗号化の欠如や不適切な実装により、機密データが保護されない脆弱性。

### リスク
- 機密情報の漏洩（パスワード、クレジットカード情報、個人情報）
- 通信の傍受
- データの改ざん
- 法的・規制上の違反

### 主な攻撃例
- 平文での機密データ保存
- 弱い暗号化アルゴリズムの使用
- HTTP での機密データ送信
- 不適切な鍵管理

### 対策

#### データ分類
1. **保存データの分類**
   - 機密情報（パスワード、個人情報、決済情報）の特定
   - データの重要度に応じた保護レベルの決定

2. **不要データの削除**
   - 必要のない機密データは保存しない
   - 保持期限を過ぎたデータの削除

#### 暗号化の実装

1. **保存データの暗号化**
   - パスワード: 適切なハッシュ関数（bcrypt, Argon2, scrypt）を使用
   - 機密情報: AES-256 などの強力な暗号化アルゴリズム
   - ソルトの使用（パスワードハッシュ）
   - 鍵の安全な管理（環境変数、Key Management Service）

2. **通信の暗号化**
   - HTTPS の強制（TLS 1.2 以上、推奨は TLS 1.3）
   - HSTS ヘッダーの設定
   - 証明書の適切な検証
   - 完全な暗号化（Mixed Content の排除）

3. **アルゴリズムの選択**
   - 古いアルゴリズムの使用禁止（MD5, SHA1, DES, RC4）
   - 業界標準の推奨アルゴリズムを使用
   - 暗号化ライブラリは実績のあるものを選択（自作しない）

4. **鍵管理**
   - 鍵をコードに埋め込まない
   - 鍵のローテーション計画
   - 鍵の安全な保管（専用の Key Management System）

#### 検証
- 定期的な暗号化設定のレビュー
- 脆弱性スキャンツールの活用
- ペネトレーションテストの実施

## A03:2021 - Injection

### 概要
信頼できない入力がコマンドやクエリの一部として解釈され、意図しない処理が実行される脆弱性。

### リスク
- データの漏洩・改ざん・削除
- システムの完全な侵害
- サービスの停止
- バックエンドシステムへの攻撃の踏み台

### 主な種類

#### SQL インジェクション
- データベースクエリの改ざん
- 全データの取得や削除
- 認証バイパス

#### OS コマンドインジェクション
- 任意のシステムコマンドの実行
- サーバーの完全な制御

#### XSS（Cross-Site Scripting）
- JavaScript の不正実行
- セッションハイジャック
- フィッシング攻撃

#### その他
- LDAP インジェクション
- XML インジェクション
- テンプレートインジェクション
- NoSQL インジェクション

### 対策

#### 基本原則
1. **入力検証**
   - すべての外部入力を検証
   - 許可リスト方式（ホワイトリスト）を優先
   - 型、長さ、形式、範囲のチェック

2. **出力エスケープ**
   - コンテキストに応じた適切なエスケープ
   - HTML, JavaScript, URL, SQL, OS コマンドなど

3. **API の安全な使用**
   - パラメータ化されたクエリ（プリペアドステートメント）
   - ORM の適切な使用
   - 安全な API の選択

#### SQL インジェクション対策
```
推奨:
- プリペアドステートメント（パラメータ化クエリ）
- ORM の使用（ただし正しく使う）
- ストアドプロシージャ（適切な場合）

非推奨:
- 文字列連結によるクエリ構築
- エスケープのみに依存
```

#### OS コマンドインジェクション対策
```
推奨:
- 可能な限りシステムコマンドを使用しない
- 専用のライブラリや API を使用
- 許可リストによる入力検証
- コマンドとパラメータの分離

非推奨:
- シェル経由でのコマンド実行
- ユーザー入力の直接使用
```

#### XSS 対策
```
推奨:
- 出力時の HTML エスケープ
- Content-Security-Policy ヘッダー
- HTTPOnly, Secure フラグ付き Cookie
- 入力の検証とサニタイゼーション

コンテキスト別エスケープ:
- HTML コンテキスト: &, <, >, ", ' のエスケープ
- JavaScript コンテキスト: JSON エンコード
- URL コンテキスト: URL エンコード
- CSS コンテキスト: CSS エスケープ
```

#### その他の対策
- 入力の長さ制限
- エラーメッセージでの情報漏洩防止
- 最小権限でのデータベース接続
- Web Application Firewall (WAF) の活用

## A04:2021 - Insecure Design

### 概要
セキュリティを考慮していない設計や、リスク分析の欠如による根本的な脆弱性。

### リスク
- 後から修正困難な構造的脆弱性
- 多層的な防御の欠如
- ビジネスロジックの悪用
- 大規模なセキュリティインシデント

### 主な問題例
- 脅威モデリングの欠如
- セキュリティ要件の未定義
- ビジネスロジックの欠陥
- レート制限やスロットリングの欠如
- 多層防御の欠如

### 対策

#### 設計段階
1. **脅威モデリング**
   - STRIDE などのフレームワークの活用
   - 攻撃シナリオの想定
   - データフロー図の作成
   - 信頼境界の明確化

2. **セキュリティ要件の定義**
   - 機能要件と並行してセキュリティ要件を定義
   - 認証・認可の仕組み
   - データ保護の要件
   - ログとモニタリングの要件

3. **セキュアアーキテクチャ**
   - 多層防御（Defense in Depth）の実装
   - 最小権限の原則
   - 責任の分離
   - フェイルセーフの設計

#### ビジネスロジックの保護
1. **リソース消費の制限**
   - レート制限の実装
   - リクエストのスロットリング
   - ページングの強制
   - タイムアウトの設定

2. **ビジネスフローの保護**
   - ステップのスキップ防止
   - トランザクションの整合性確保
   - リプレイ攻撃の防止
   - 同時実行制御

3. **異常検知**
   - 不正なパターンの検出
   - 自動化された攻撃の検知
   - アノマリー検出

#### 開発プロセス
- セキュアな SDLC の採用
- デザインレビューの実施
- セキュリティテストの統合
- 定期的なペネトレーションテスト

## A05:2021 - Security Misconfiguration

### 概要
不適切なセキュリティ設定や、デフォルト設定の放置による脆弱性。

### リスク
- 不正アクセスの許可
- 機密情報の漏洩
- システムの侵害
- サービスの停止

### 主な問題例
- デフォルト認証情報の使用
- 不要な機能の有効化
- エラーメッセージでの詳細情報の表示
- セキュリティヘッダーの欠如
- 古いバージョンの使用
- 不適切な権限設定

### 対策

#### サーバー・インフラ設定
1. **最小限の構成**
   - 不要な機能・サービスの無効化
   - 未使用のポートのクローズ
   - サンプルアプリケーションの削除
   - 開発ツールの本番環境からの除外

2. **デフォルト設定の変更**
   - デフォルトパスワードの変更
   - デフォルトアカウントの無効化または削除
   - 管理画面の URL 変更またはアクセス制限

3. **権限管理**
   - ファイル・ディレクトリの適切な権限設定
   - 最小権限でのプロセス実行
   - 書き込み権限の最小化

#### アプリケーション設定
1. **エラーハンドリング**
   - 本番環境での詳細エラーメッセージの非表示
   - スタックトレースの非表示
   - カスタムエラーページの使用
   - 適切なログ記録

2. **セキュリティヘッダー**
   - Content-Security-Policy
   - Strict-Transport-Security
   - X-Content-Type-Options
   - X-Frame-Options
   - Referrer-Policy

3. **デバッグ・開発機能**
   - 本番環境でのデバッグモード無効化
   - ソースマップの非公開
   - 開発用エンドポイントの削除

#### 環境管理
1. **環境分離**
   - 開発・ステージング・本番の明確な分離
   - 環境ごとの異なる認証情報
   - 本番データの開発環境での使用禁止

2. **更新管理**
   - 定期的なパッチ適用
   - セキュリティアップデートの迅速な適用
   - バージョン管理

3. **設定管理**
   - Infrastructure as Code の活用
   - 設定の自動化
   - 定期的な設定レビュー
   - 設定の監査

#### 継続的な改善
- 定期的なセキュリティスキャン
- 設定のベースライン作成
- コンプライアンスチェック
- 自動化されたセキュリティテスト

## A06:2021 - Vulnerable and Outdated Components

### 概要
既知の脆弱性を含むライブラリやフレームワークの使用による脆弱性。

### リスク
- 既知の脆弱性の悪用
- リモートコード実行
- データ漏洩
- システムの完全な侵害

### 主な問題例
- 古いバージョンのライブラリ使用
- サポート終了したコンポーネントの使用
- 脆弱性の未パッチ
- 依存関係の管理不足
- 不要な依存関係の含有

### 対策

#### 依存関係の管理
1. **インベントリ管理**
   - すべての依存関係のリスト作成
   - 直接・間接依存関係の把握
   - バージョンの記録
   - Software Bill of Materials (SBOM) の作成

2. **信頼できるソースからの取得**
   - 公式リポジトリの使用
   - パッケージの整合性検証（チェックサム、署名）
   - プライベートリポジトリの活用

3. **最小限の依存関係**
   - 不要な依存関係の削除
   - 機能が重複するライブラリの整理
   - 軽量な代替の検討

#### 脆弱性管理
1. **継続的な監視**
   - 脆弱性データベースの監視（NVD, CVE）
   - セキュリティアドバイザリの購読
   - 自動スキャンツールの活用

2. **脆弱性スキャン**
   - CI/CD パイプラインへの統合
   - 定期的なスキャン実行
   - 脆弱性の重要度評価
   - 誤検知の管理

3. **更新戦略**
   - 定期的な更新スケジュール
   - セキュリティパッチの優先適用
   - 更新前のテスト
   - ロールバック計画

#### 開発プロセス
1. **依存関係の追加**
   - 新規依存関係の事前評価
   - メンテナンス状況の確認
   - ライセンスの確認
   - コミュニティの活発さの確認

2. **ロックファイルの使用**
   - 依存関係のバージョン固定
   - 再現可能なビルド
   - 予期しない更新の防止

3. **廃止されたコンポーネントの対応**
   - サポート終了の事前把握
   - 代替ソリューションの検討
   - 移行計画の策定

#### ツールとプラクティス
```
推奨ツール例:
- 脆弱性スキャナー（言語・エコシステム固有）
- 依存関係管理ツール
- SBOM 生成ツール
- CI/CD 統合

定期的な実施:
- 依存関係の更新（月次など）
- 脆弱性スキャン（毎ビルド）
- セキュリティアドバイザリの確認（週次）
- 依存関係の棚卸し（四半期）
```

## A07:2021 - Identification and Authentication Failures

### 概要
認証機能の不備により、ユーザーの身元確認が適切に行われない脆弱性。

### リスク
- アカウントの乗っ取り
- なりすまし
- 機密情報への不正アクセス
- セッションハイジャック

### 主な問題例
- 弱いパスワードの許可
- ブルートフォース攻撃への対策不足
- 資格情報の平文保存
- セッション管理の不備
- 多要素認証の欠如

### 対策

#### パスワード管理
1. **パスワードポリシー**
   - 最小文字数の設定（推奨: 12 文字以上）
   - 複雑性要件（大小文字、数字、記号）
   - よく使われるパスワードのブロック
   - パスワード履歴のチェック

2. **パスワード保存**
   - 適切なハッシュ関数の使用（bcrypt, Argon2, scrypt）
   - ソルトの使用（ユーザーごとにユニーク）
   - ペッパーの使用（オプション）
   - 平文やリバーシブルな暗号化の禁止

3. **パスワードリセット**
   - 安全なトークンの生成（暗号学的に安全な乱数）
   - トークンの有効期限設定
   - 一度きりの使用
   - 既存セッションの無効化

#### 認証プロセス
1. **ログイン保護**
   - レート制限の実装
   - アカウントロックアウト（一時的）
   - CAPTCHA の使用（過度な試行時）
   - タイミング攻撃への対策（定時間応答）

2. **多要素認証（MFA）**
   - 重要な操作には MFA を要求
   - 複数の認証要素の組み合わせ
   - リカバリーコードの提供
   - MFA バイパスの防止

3. **資格情報の扱い**
   - HTTPS での送信強制
   - URL パラメータに資格情報を含めない
   - ログに資格情報を記録しない
   - オートコンプリートの適切な制御

#### セッション管理
1. **セッション生成**
   - 暗号学的に安全なセッション ID
   - 十分な長さとランダム性
   - ログイン成功時に新しいセッション ID を発行
   - ログアウト時のセッション無効化

2. **セッション保護**
   - Cookie の適切な設定（HttpOnly, Secure, SameSite）
   - セッションタイムアウトの設定
   - アイドルタイムアウトの実装
   - 絶対タイムアウトの実装

3. **セッション固定化への対策**
   - 認証前後でセッション ID を変更
   - セッション ID の推測困難性
   - セッション ID の URL への非含有

#### その他の対策
1. **アカウント登録**
   - メールアドレス検証
   - 重複アカウントのチェック
   - 登録時のレート制限
   - 自動化防止

2. **ログとモニタリング**
   - ログイン試行の記録
   - 異常なパターンの検知
   - 地理的位置の変化の検知
   - 複数デバイスからの同時ログインの検知

3. **ソーシャルエンジニアリング対策**
   - セキュリティ質問の適切な使用
   - ユーザー教育
   - 疑わしいアクティビティの通知

## A08:2021 - Software and Data Integrity Failures

### 概要
コードやインフラの完全性検証の欠如により、不正な更新や改ざんを検知できない脆弱性。

### リスク
- 不正なコードの実行
- サプライチェーン攻撃
- データの改ざん
- マルウェアの混入

### 主な問題例
- 未署名の更新
- 信頼できないソースからのコード取得
- CI/CD パイプラインのセキュリティ不備
- 整合性チェックの欠如
- 安全でないデシリアライゼーション

### 対策

#### コードの完全性
1. **ソースコード管理**
   - バージョン管理システムの使用
   - コミットの署名
   - コードレビューの義務化
   - ブランチ保護ルール

2. **ビルドプロセス**
   - 再現可能なビルド
   - ビルド成果物の署名
   - ビルド環境の隔離

## A09:2021 - Security Logging and Monitoring Failures

### 概要
不適切なログ記録やモニタリングにより、セキュリティインシデントの検知が遅れる脆弱性。

### リスク
- 攻撃の検知遅延
- インシデント対応の遅れ
- 被害の拡大
- 原因究明の困難
- コンプライアンス違反

### 主な問題例
- ログの記録不足
- ログの監視不足
- アラート機能の欠如
- ログの改ざん可能性
- ログの保存期間不足

### 対策

#### ログ記録
1. **記録すべきイベント**
   - 認証の成功・失敗
   - アクセス制御の失敗
   - 入力検証の失敗
   - 重要な操作（作成、変更、削除）
   - 設定変更
   - 権限の変更
   - セッションの開始・終了
   - エラーと例外

2. **ログの内容**
   - タイムスタンプ（UTC 推奨）
   - ユーザー識別子
   - IP アドレス
   - イベントタイプ
   - リソース識別子
   - 操作の結果
   - 重要度レベル

3. **記録してはいけない情報**
   - パスワード
   - セッション ID
   - クレジットカード情報
   - 個人識別可能情報（PII）（必要最小限に）
   - API キー・トークン

#### ログの管理
1. **ログの保護**
   - アクセス制御（読み取り専用）
   - 改ざん防止（追記専用）
   - 暗号化（保存時・転送時）
   - バックアップ

2. **ログの保存**
   - 適切な保存期間の設定
   - 法的要件の遵守
   - ローテーション
   - アーカイブ

3. **ログの集約**
   - 中央ログ管理システムの使用
   - 複数のソースからの収集
   - 相関分析の実施
   - タイムスタンプの同期

#### モニタリングとアラート
1. **リアルタイム監視**
   - 疑わしいパターンの検知
   - 閾値ベースのアラート
   - 異常検知
   - 自動化された対応

2. **アラート設定**
   - 重要度に応じたアラート
   - 適切な通知先の設定
   - エスカレーションルール
   - アラート疲れの防止

3. **監視対象**
   - ログイン試行の異常
   - 権限昇格の試み
   - 大量のリクエスト
   - エラー率の上昇
   - リソース使用量の異常
   - 既知の攻撃パターン

#### インシデント対応
1. **準備**
   - インシデント対応計画の策定
   - 対応チームの編成
   - 連絡体制の確立
   - ツールの準備

2. **検知と分析**
   - ログの定期的なレビュー
   - セキュリティイベントの分析
   - 相関分析
   - 脅威インテリジェンスの活用

3. **対応と復旧**
   - 迅速な初動対応
   - 影響範囲の特定
   - 封じ込めと根絶
   - システムの復旧
   - 事後レビュー

#### コンプライアンス
- 業界標準への準拠（PCI DSS, GDPR など）
- 監査証跡の確保
- 定期的な監査
- 報告義務の遵守

## A10:2021 - Server-Side Request Forgery (SSRF)

### 概要
サーバーが外部リソースを取得する際、URL の検証不足により、攻撃者が意図しないリクエストを送信させられる脆弱性。

### リスク
- 内部ネットワークのスキャン
- 内部サービスへのアクセス
- クラウドメタデータの窃取
- 認証情報の漏洩
- リモートコード実行

### 主な攻撃例
- クラウドメタデータエンドポイントへのアクセス（AWS: 169.254.169.254）
- 内部 IP アドレスへのアクセス（127.0.0.1, 10.0.0.0/8）
- ファイルスキームの悪用（file:///etc/passwd）
- DNS リバインディング攻撃
- リダイレクトの悪用

### 対策

#### 入力検証
1. **URL の検証**
   - 許可リスト方式（ホワイトリスト）の採用
   - URL スキームの制限（http, https のみ）
   - ドメインの検証
   - IP アドレスの検証

2. **禁止すべきアクセス先**
   - プライベート IP アドレス範囲
     - 127.0.0.0/8（ループバック）
     - 10.0.0.0/8（プライベート）
     - 172.16.0.0/12（プライベート）
     - 192.168.0.0/16（プライベート）
     - 169.254.0.0/16（リンクローカル）
   - ローカルホスト（localhost, 127.0.0.1, ::1）
   - クラウドメタデータエンドポイント
   - 内部ドメイン

3. **DNS 検証**
   - DNS 解決後の IP アドレス検証
   - DNS リバインディング対策
   - 短い TTL の拒否
   - 複数回の解決と検証

#### ネットワークレベルの対策
1. **ネットワーク分離**
   - DMZ の使用
   - ファイアウォールルール
   - セグメンテーション
   - 最小権限のアクセス

2. **アウトバウンド制御**
   - プロキシの使用
   - ホワイトリストベースのアクセス
   - VPN の活用
   - アウトバウンドファイアウォール

3. **内部サービスの保護**
   - 認証の強化
   - IP ベースの制限
   - ネットワークポリシー
   - サービスメッシュの活用

#### アプリケーションレベルの対策
1. **リダイレクトの処理**
   - リダイレクトの無効化または制限
   - リダイレクト先の検証
   - リダイレクト回数の制限
   - オープンリダイレクトの防止

2. **レスポンスの処理**
   - レスポンスサイズの制限
   - コンテンツタイプの検証
   - タイムアウトの設定
   - エラー情報の制限

3. **機能の制限**
   - 不要な URL フェッチ機能の削除
   - 代替実装の検討
   - サーバーサイドでの処理を避ける
   - サンドボックス化

#### クラウド環境の対策
1. **メタデータサービスの保護**
   - IMDSv2 の使用（AWS）
   - メタデータエンドポイントへのアクセス制限
   - ホップリミットの設定
   - 認証の要求

2. **IAM ロールの管理**
   - 最小権限の原則
   - 一時的な認証情報
   - 定期的なローテーション
   - アクセスログの監視

#### モニタリング
- 異常なアウトバウンドリクエストの検知
- 内部 IP へのアクセス試行の記録
- メタデータエンドポイントへのアクセス監視
- DNS クエリの監視

## まとめ

OWASP Top 10 は Web アプリケーションの最も重大なセキュリティリスクをまとめたものです。これらの対策を実装することで、多くの一般的な攻撃から保護できます。

### 共通原則
1. **多層防御**: 単一の対策に依存しない
2. **最小権限**: 必要最小限のアクセス権限のみ付与
3. **デフォルト拒否**: すべてを拒否し、明示的に許可
4. **フェイルセーフ**: エラー時は安全側に倒す
5. **責任の分離**: 機能と権限を分離

### 継続的な取り組み
- 定期的なセキュリティレビュー
- 脆弱性スキャンの実施
- セキュリティテストの自動化
- 開発者教育
- インシデント対応計画の策定と訓練
