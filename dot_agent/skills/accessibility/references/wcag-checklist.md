## はじめに

このドキュメントは、WCAG 2.1のレベルAおよびAAの達成基準を整理したチェックリストです。各基準について、説明、確認方法、よくある失敗例を記載しています。

**適合レベルについて:**
- **レベルA:** 最低限の適合レベル
- **レベルAA:** 推奨される適合レベル(多くの組織が目標とする)
- **レベルAAA:** 最高の適合レベル(このドキュメントでは扱いません)

## 知覚可能 (Perceivable)

ユーザーがコンテンツを知覚できるようにする必要があります。

### 1.1 テキストによる代替

#### 1.1.1 非テキストコンテンツ (レベルA)

**説明:**
画像、アイコン、ボタンなど、すべての非テキストコンテンツには、同等の目的を果たすテキストによる代替を提供する必要があります。

**確認方法:**
- 画像に適切な`alt`属性が設定されているか確認
- 装飾的な画像には`alt=""`(空の代替テキスト)が設定されているか確認
- アイコンボタンに`aria-label`または視覚的に隠されたテキストがあるか確認
- CAPTCHAには代替手段(音声版など)があるか確認
- スクリーンリーダーで確認し、意味が伝わるか検証

**よくある失敗例:**
- `<img src="logo.png">`のように`alt`属性が欠落している
- `<img src="chart.png" alt="chart">`のように内容を説明していない
- `<img src="decorative.png" alt="decorative image">`のように装飾画像に不要な説明がある
- アイコンのみのボタンにラベルがない
- 情報を伝える画像テキストを`alt`で説明していない

### 1.2 時間依存メディア

#### 1.2.1 音声のみ及び映像のみ (収録済) (レベルA)

**説明:**
収録済みの音声のみ、または映像のみのコンテンツには、同等の情報を提供する代替コンテンツが必要です。

**確認方法:**
- 音声のみのコンテンツ(ポッドキャストなど)にテキスト版またはトランスクリプトがあるか確認
- 映像のみのコンテンツ(サイレント動画など)に音声トラックまたはテキスト説明があるか確認
- 代替コンテンツが同等の情報を提供しているか確認

**よくある失敗例:**
- ポッドキャストにトランスクリプトがない
- 説明動画(音声なし)に代替説明がない
- 代替テキストが不完全で重要な情報が欠けている

#### 1.2.2 キャプション (収録済) (レベルA)

**説明:**
収録済みの動画コンテンツには、同期したキャプション(字幕)を提供する必要があります。

**確認方法:**
- 動画にキャプションが表示されるか確認
- キャプションが音声と同期しているか確認
- 話者の識別、効果音、重要な非音声情報が含まれているか確認
- キャプションのオン/オフが可能か確認

**よくある失敗例:**
- キャプションが提供されていない
- キャプションと音声のタイミングがずれている
- 話者が区別されていない
- 重要な効果音(ドアが閉まる音など)が記述されていない

#### 1.2.3 音声解説、又はメディアに対する代替 (収録済) (レベルA)

**説明:**
収録済みの動画コンテンツには、音声解説またはテキストによる代替を提供する必要があります。

**確認方法:**
- 視覚的な情報(アクション、シーンの変化など)が音声で説明されているか確認
- または、完全なテキストトランスクリプト(視覚情報を含む)が提供されているか確認

**よくある失敗例:**
- 視覚的にのみ示される重要な情報に音声解説がない
- トランスクリプトに視覚情報が含まれていない

#### 1.2.4 キャプション (ライブ) (レベルAA)

**説明:**
ライブ配信の音声コンテンツには、リアルタイムのキャプションを提供する必要があります。

**確認方法:**
- ライブ配信でキャプションが利用可能か確認
- キャプションがリアルタイムで表示されるか確認
- 遅延が許容範囲内か確認

**よくある失敗例:**
- ライブウェビナーにキャプションがない
- 自動生成キャプションの精度が低すぎる

#### 1.2.5 音声解説 (収録済) (レベルAA)

**説明:**
収録済みの動画コンテンツには、音声解説を提供する必要があります(1.2.3の要件をより厳密にしたもの)。

**確認方法:**
- 動画に音声解説トラックがあるか確認
- 視覚的な情報が適切に説明されているか確認
- 音声解説のオン/オフが可能か確認

**よくある失敗例:**
- 音声解説が提供されていない
- 音声解説が不完全で重要な視覚情報を伝えていない

### 1.3 適応可能

#### 1.3.1 情報及び関係性 (レベルA)

**説明:**
表現を通じて伝達される情報、構造、および関係性は、プログラムによる解釈が可能であるか、テキストで利用可能である必要があります。

**確認方法:**
- 見出しに`<h1>`～`<h6>`タグが使用されているか確認
- リストに`<ul>`, `<ol>`, `<li>`が使用されているか確認
- テーブルに`<th>`, `<caption>`, `scope`属性が適切に使用されているか確認
- フォームのラベルが`<label>`または`aria-label`で関連付けられているか確認
- CSSで視覚的に表現されている関係性がHTMLでも表現されているか確認
- スクリーンリーダーで構造が理解できるか確認

**よくある失敗例:**
- 見出しを`<div>`に大きなフォントサイズを適用して表現している
- `<div>`のリストをCSSで装飾して箇条書きに見せている
- テーブルの見出しが`<th>`でマークアップされていない
- フォームコントロールとラベルが関連付けられていない
- 視覚的なグループ化が意味的にマークアップされていない

#### 1.3.2 意味のある順序 (レベルA)

**説明:**
コンテンツが提示される順序が意味に影響する場合、正しい読み上げ順序がプログラムによって解釈可能である必要があります。

**確認方法:**
- HTMLのソース順序が論理的な読み上げ順序になっているか確認
- CSSで視覚的な順序を変更している場合、スクリーンリーダーの読み上げ順序が適切か確認
- Tabキーでフォーカス順序を確認
- スクリーンリーダーで内容を確認

**よくある失敗例:**
- CSSの`position: absolute`や`float`で配置を変更し、視覚順序とDOM順序が異なる
- Flexboxやグridの`order`プロパティで順序を変更し、読み上げ順序が不自然になっている
- サイドバーがHTMLでメインコンテンツより前に配置されている(モバイル対応で問題になる)

#### 1.3.3 感覚的な特徴 (レベルA)

**説明:**
コンテンツを理解し操作するための説明は、形、色、サイズ、視覚的な位置、方向、音のみに依存してはいけません。

**確認方法:**
- 「右側のボタンをクリック」のような位置のみの指示がないか確認
- 「赤いボタンをクリック」のような色のみの指示がないか確認
- 「大きなボタンをクリック」のようなサイズのみの指示がないか確認
- アイコン形状のみでなく、テキストラベルもあるか確認

**よくある失敗例:**
- 「右のフォームに入力してください」(位置のみ)
- 「赤い項目は必須です」(色のみ)
- 「大きなアイコンをクリックして開始」(サイズのみ)
- 音声のみの通知やアラート

#### 1.3.4 表示の向き (レベルAA)

**説明:**
コンテンツは、縦向きまたは横向きのどちらか一方の表示に制限されてはいけません(特定の向きが必須の場合を除く)。

**確認方法:**
- 画面を回転させてもコンテンツが利用可能か確認
- CSSで`orientation`を固定していないか確認
- JavaScriptで向きを制限していないか確認

**よくある失敗例:**
- CSSで`@media (orientation: portrait) { transform: rotate(90deg); }`のように強制している
- JavaScriptで横向きのみに制限している
- 「デバイスを回転してください」というメッセージのみで機能しない

#### 1.3.5 入力目的の特定 (レベルAA)

**説明:**
ユーザー情報を収集する入力フィールドの目的は、プログラムによって特定可能である必要があります。

**確認方法:**
- フォームフィールドに適切な`autocomplete`属性が設定されているか確認
- 名前、メール、電話番号、住所などの一般的なフィールドに標準的な値が使用されているか確認
- HTML5の`type`属性が適切に使用されているか確認

**よくある失敗例:**
- 名前フィールドに`autocomplete`属性がない
- `<input type="text" name="email">`で`autocomplete="email"`を使用していない
- 住所フィールドに適切な`autocomplete`値(street-address, postal-codeなど)が設定されていない

### 1.4 判別可能

#### 1.4.1 色の使用 (レベルA)

**説明:**
色が、情報を伝える、動作を示す、応答を促す、または視覚的な要素を区別する唯一の視覚的手段として使用されてはいけません。

**確認方法:**
- リンクが色だけでなく下線や太字でも区別されているか確認
- エラーメッセージが色だけでなくアイコンやテキストでも示されているか確認
- グラフや図表が色だけでなくパターンやラベルでも区別されているか確認
- 必須フィールドが色だけでなくアスタリスクやテキストでも示されているか確認
- グレースケール表示で情報が失われないか確認

**よくある失敗例:**
- リンクが色のみで区別され、下線がない
- エラーフィールドが赤枠のみで示され、エラーアイコンやテキストがない
- 「緑は承認、赤は拒否」のように色のみで状態を表現
- グラフの系列が色のみで区別され、パターンやラベルがない

#### 1.4.2 音声の制御 (レベルA)

**説明:**
3秒以上自動再生される音声には、一時停止、停止、または音量調整の仕組みが必要です。

**確認方法:**
- 自動再生される音声に一時停止/停止ボタンがあるか確認
- 音量調整が可能か確認
- 音声コントロールがキーボードでアクセス可能か確認

**よくある失敗例:**
- 背景音楽が自動再生され、停止できない
- 動画が自動再生され、音声をミュートできない
- 音声コントロールがマウスでのみ操作可能

#### 1.4.3 最低限のコントラスト (レベルAA)

**説明:**
テキストと背景のコントラスト比は、通常のテキストで4.5:1以上、大きなテキスト(18pt以上または14pt以上の太字)で3:1以上である必要があります。

**確認方法:**
- カラーコントラストチェッカーツールで確認(例: WebAIM Contrast Checker, Chrome DevTools)
- すべてのテキスト、リンク、ボタンのコントラストを確認
- ホバーやフォーカス状態のコントラストも確認
- プレースホルダーテキストのコントラストも確認

**よくある失敗例:**
- 薄いグレーのテキストが白背景に使用されている(コントラスト比3:1など)
- リンクの色が薄くてコントラスト不足
- ボタンのテキストと背景のコントラストが不十分
- プレースホルダーが読みにくい

#### 1.4.4 テキストのサイズ変更 (レベルAA)

**説明:**
キャプションや画像化されたテキストを除いて、テキストは支援技術なしで200%まで拡大でき、コンテンツや機能が失われないようにする必要があります。

**確認方法:**
- ブラウザのズーム機能で200%に拡大して確認
- テキストが切れたり重なったりしないか確認
- 水平スクロールが発生しないか確認(縦書きを除く)
- すべての機能が利用可能か確認

**よくある失敗例:**
- 固定幅のコンテナでテキストが切れる
- フォントサイズが`px`で固定されており、拡大できない
- 拡大時にコンテンツが重なる
- ナビゲーションメニューが崩れて使えなくなる

#### 1.4.5 画像化された文字 (レベルAA)

**説明:**
テキストで同じ視覚的な表現が可能な場合は、画像化されたテキストではなく、テキストを使用する必要があります。

**確認方法:**
- ロゴやブランド名を除いて、テキストが画像として表示されていないか確認
- CSSで同じスタイルを実現できないか検討
- 必要な場合は、画像に適切な代替テキストがあるか確認

**よくある失敗例:**
- 見出しやボタンのテキストが画像として作成されている
- 特殊フォントを使用するためにテキストを画像化している
- CSSで実現可能な装飾テキストを画像にしている

#### 1.4.10 リフロー (レベルAA)

**説明:**
コンテンツは、情報や機能を失うことなく、320 CSS ピクセルの幅(縦書きの場合は高さ)で表示できる必要があります。2次元レイアウトが必要なコンテンツ(地図、図表など)は除きます。

**確認方法:**
- ブラウザウィンドウを320px幅にリサイズして確認
- 水平スクロールが必要ないか確認(縦書きの場合は垂直スクロール)
- すべてのコンテンツと機能が利用可能か確認
- 400%ズーム時に問題がないか確認

**よくある失敗例:**
- 固定幅レイアウトで水平スクロールが必要になる
- レスポンシブデザインが実装されていない
- 狭い画面でコンテンツが隠れる
- テーブルが横スクロールなしで表示できない(2次元レイアウトとして許容される場合もある)

#### 1.4.11 非テキストのコントラスト (レベルAA)

**説明:**
ユーザインターフェースコンポーネントおよび画像化された情報の視覚的な表現は、隣接する色と3:1以上のコントラスト比を持つ必要があります。

**確認方法:**
- ボタンの境界線と背景のコントラストを確認
- フォーム入力欄の境界線と背景のコントラストを確認
- アイコンと背景のコントラストを確認
- フォーカスインジケーターのコントラストを確認
- グラフやチャートの要素のコントラストを確認
- カラーコントラストチェッカーで確認

**よくある失敗例:**
- 薄いグレーの境界線が白背景のフォーム入力欄
- 薄い色のアイコンが背景と区別しにくい
- フォーカスインジケーターが薄くて見えにくい
- グラフの線や領域のコントラストが不十分

#### 1.4.12 テキストの間隔 (レベルAA)

**説明:**
ユーザーが以下のテキスト間隔を調整しても、コンテンツや機能が失われないようにする必要があります:
- 行の高さを1.5倍以上
- 段落の間隔を2倍以上
- 文字間隔を0.12倍以上
- 単語間隔を0.16倍以上

**確認方法:**
- ブラウザ拡張機能(Text Spacing Bookmarkletなど)で上記の間隔を適用して確認
- テキストが切れたり重なったりしないか確認
- コンテンツが隠れないか確認
- ボタンやリンクが使用可能か確認

**よくある失敗例:**
- 固定高さのコンテナでテキストが切れる
- `line-height`や`letter-spacing`を`!important`で固定している
- テキスト間隔を増やすとレイアウトが崩れる

#### 1.4.13 ホバー又はフォーカスで表示されるコンテンツ (レベルAA)

**説明:**
ホバーやフォーカスで表示されるコンテンツ(ツールチップ、サブメニューなど)は、以下の条件を満たす必要があります:
- **却下可能:** Escキーなどでポインターを移動せずに閉じられる
- **ホバー可能:** ポインターをコンテンツ上に移動してもコンテンツが消えない
- **持続的:** ユーザーが閉じるか、関連性がなくなるまで表示される

**確認方法:**
- ツールチップやサブメニューをホバー/フォーカスで表示して確認
- Escキーで閉じられるか確認
- ポインターをツールチップ上に移動できるか確認
- 意図せず消えないか確認
- フォーカスを移動したときの動作を確認

**よくある失敗例:**
- ツールチップがEscキーで閉じられない
- ツールチップ上にポインターを移動すると消える
- サブメニューがすぐに消えてクリックできない
- ホバー時のみ表示されるコンテンツで、キーボードでアクセスできない

## 操作可能 (Operable)

ユーザーがインターフェースコンポーネントを操作できるようにする必要があります。

### 2.1 キーボードアクセシブル

#### 2.1.1 キーボード (レベルA)

**説明:**
すべての機能がキーボードインターフェースのみで操作可能である必要があります(タイミングに依存しない限り)。

**確認方法:**
- マウスを使用せずにTabキーですべてのインタラクティブ要素に到達できるか確認
- Enter/Spaceキーでボタンやリンクを活性化できるか確認
- 矢印キーでドロップダウンやタブを操作できるか確認
- Escキーでモーダルやメニューを閉じられるか確認
- カスタムウィジェットがキーボードで操作可能か確認

**よくある失敗例:**
- `<div onclick="...">`のようなキーボードでアクセスできない要素
- カスタムドロップダウンがキーボードで開けない
- ドラッグ&ドロップのみで並び替えができる
- ホバーのみで表示されるメニューがキーボードでアクセスできない
- `tabindex="-1"`で誤ってフォーカス可能な要素を除外している

#### 2.1.2 キーボードトラップなし (レベルA)

**説明:**
キーボードでフォーカスをコンポーネントに移動できる場合、キーボードのみでフォーカスを離れることができる必要があります。

**確認方法:**
- タブナビゲーションでページ全体を移動できるか確認
- モーダルやウィジェット内でフォーカスが閉じ込められていないか確認
- フォーカストラップが実装されている場合、終了方法が明確か確認
- 埋め込みコンテンツ(iframe、プラグインなど)からフォーカスを離れられるか確認

**よくある失敗例:**
- モーダルを開いた際にTabキーで背景コンテンツにフォーカスが移動できない、かつEscで閉じられない
- カスタムウィジェットからTabキーで抜けられない
- iframeにフォーカスが入ると抜けられない
- JavaScriptでフォーカスを強制的に戻している

#### 2.1.4 文字キーのショートカット (レベルA)

**説明:**
単一の文字キーのショートカットがある場合、それをオフにする、再割り当てする、またはフォーカス時のみ有効にする方法が必要です。

**確認方法:**
- 文字キーのショートカットが実装されているか確認
- ショートカットを無効化または変更する設定があるか確認
- フォームフィールドにフォーカスがある時、ショートカットが無効になるか確認

**よくある失敗例:**
- 文字キー(例: "S"で検索、"N"で次へ)が常に有効で無効化できない
- 音声入力ユーザーが誤ってショートカットをトリガーする
- フォーム入力中にショートカットが実行される

---

### 2.2 十分な時間

#### 2.2.1 タイミング調整可能 (レベルA)

**説明:**
時間制限がある場合、ユーザーが制限を解除、調整、または延長できる必要があります(リアルタイムイベントや期限が必須の場合などを除く)。

**確認方法:**
- セッションタイムアウト前に警告が表示されるか確認
- タイムアウトを延長するオプションがあるか確認
- 時間制限を無効化または調整する設定があるか確認
- 少なくとも20秒前に警告が表示されるか確認

**よくある失敗例:**
- セッションが警告なしで突然タイムアウトする
- カウントダウンタイマーがあるが延長できない
- タイムアウト警告が短すぎて対応できない
- 自動ログアウトを無効化できない

#### 2.2.2 一時停止、停止、非表示 (レベルA)

**説明:**
自動的に動く、点滅する、スクロールする、または自動更新される情報は、一時停止、停止、または非表示にできる必要があります(5秒以上続く場合)。

**確認方法:**
- カルーセルに一時停止ボタンがあるか確認
- 自動スクロールのニュースティッカーを停止できるか確認
- 自動更新コンテンツを一時停止できるか確認
- アニメーションを停止するコントロールがあるか確認
- マウスホバーで一時停止する場合、キーボードでも一時停止できるか確認

**よくある失敗例:**
- カルーセルが自動で切り替わり、停止できない
- ニュースティッカーが常にスクロールしている
- 自動更新される株価情報を停止できない
- アニメーションバナーに停止ボタンがない

---

### 2.3 発作の防止

#### 2.3.1 3回の閃光、又は閾値以下 (レベルA)

**説明:**
1秒間に3回以上閃光を放つコンテンツを含めてはいけません(十分に小さい、低コントラストの場合を除く)。

**確認方法:**
- アニメーションや動画に高速で点滅する要素がないか確認
- 自動再生される広告やエフェクトをチェック
- ゲームやインタラクティブコンテンツの閃光を確認

**よくある失敗例:**
- 高速で点滅するアニメーション広告
- ストロボライトのようなエフェクト
- ゲーム内の爆発エフェクトで激しい閃光

---

### 2.4 ナビゲーション可能

#### 2.4.1 ブロックスキップ (レベルA)

**説明:**
複数のページで繰り返されるコンテンツブロック(ナビゲーション、ヘッダーなど)をスキップする仕組みが必要です。

**確認方法:**
- 「メインコンテンツへスキップ」リンクがページ先頭にあるか確認
- スキップリンクがキーボードでアクセス可能か確認(フォーカス時に表示されるか)
- ARIAランドマーク(`main`, `navigation`, `banner`など)が適切に使用されているか確認
- 見出し構造が適切で、スクリーンリーダーでナビゲートしやすいか確認

**よくある失敗例:**
- スキップリンクがない
- スキップリンクが`display:none`で完全に隠されている(フォーカス時も表示されない)
- スキップリンクのリンク先が不適切
- ARIAランドマークが設定されていない

#### 2.4.2 ページタイトル (レベルA)

**説明:**
ウェブページには、トピックや目的を説明するタイトルが必要です。

**確認方法:**
- すべてのページに`<title>`要素があるか確認
- タイトルがページの内容を適切に説明しているか確認
- タイトルが一意で他のページと区別できるか確認
- タイトルが簡潔で意味のあるものか確認

**よくある失敗例:**
- タイトルがない(`<title></title>`)
- すべてのページで同じタイトル(「ホームページ」など)
- タイトルが無意味(「Untitled Document」など)
- タイトルが長すぎて主要な情報が最後にある

#### 2.4.3 フォーカス順序 (レベルA)

**説明:**
ページがキーボードで順次ナビゲート可能な場合、フォーカス順序が意味や操作性を保持する必要があります。

**確認方法:**
- Tabキーでフォーカス順序を確認
- フォーカス順序が視覚的な順序と一致しているか確認
- 論理的な流れでコンテンツを読めるか確認
- モーダルやメニューが開いた時のフォーカス管理を確認

**よくある失敗例:**
- `tabindex`の正の値(1, 2, 3...)で不自然な順序にしている
- CSSで配置を変更し、視覚順序とタブ順序が異なる
- モーダルを開いても背景にフォーカスが残る
- 動的に追加されたコンテンツのフォーカス管理が不適切

#### 2.4.4 リンクの目的 (コンテキスト内) (レベルA)

**説明:**
各リンクの目的が、リンクテキスト単独で、またはプログラムで決定可能なリンクのコンテキストと組み合わせて特定できる必要があります。

**確認方法:**
- リンクテキストが目的地や機能を説明しているか確認
- 「こちら」「もっと読む」などの曖昧なリンクがないか確認
- 画像リンクに適切な代替テキストがあるか確認
- アイコンのみのリンクに`aria-label`があるか確認
- スクリーンリーダーでリンク一覧を確認

**よくある失敗例:**
- 「こちらをクリック」「詳細」のみのリンクテキスト
- 同じリンクテキストが異なる目的地へのリンクに使用されている
- アイコンリンクにラベルがない
- 「もっと読む」リンクが複数あり、何についてか不明

#### 2.4.5 複数の手段 (レベルAA)

**説明:**
ウェブページを見つけるための複数の手段(ナビゲーション、サイトマップ、検索など)が必要です。

**確認方法:**
- グローバルナビゲーションメニューがあるか確認
- サイトマップページがあるか確認
- 検索機能があるか確認
- パンくずリストがあるか確認
- 少なくとも2つの方法でページにアクセスできるか確認

**よくある失敗例:**
- ナビゲーションメニューのみでサイトマップも検索もない
- 深い階層のページへのアクセス手段が1つしかない

#### 2.4.6 見出し及びラベル (レベルAA)

**説明:**
見出しやラベルは、トピックや目的を説明する必要があります。

**確認方法:**
- 見出しがセクションの内容を適切に説明しているか確認
- フォームラベルがフィールドの目的を明確に説明しているか確認
- 見出しが具体的で説明的か確認
- ラベルが簡潔で理解しやすいか確認

**よくある失敗例:**
- 「情報」「詳細」のような曖昧な見出し
- 「テキスト1」「入力」のような無意味なラベル
- 見出しがなく、コンテンツが構造化されていない

#### 2.4.7 フォーカスの可視化 (レベルAA)

**説明:**
キーボード操作可能なユーザインターフェースには、フォーカスインジケーターが視覚的に確認できる必要があります。

**確認方法:**
- Tabキーで移動した際にフォーカスが視覚的に明確か確認
- すべてのインタラクティブ要素でフォーカスインジケーターが表示されるか確認
- フォーカスインジケーターのコントラストが十分か確認
- カスタムスタイルでブラウザデフォルトのフォーカスを削除している場合、代替スタイルがあるか確認

**よくある失敗例:**
- `outline: none`でフォーカスインジケーターを削除し、代替を提供していない
- フォーカスインジケーターが薄すぎて見えない
- 背景色と同じ色のフォーカスインジケーター
- カスタムウィジェットでフォーカススタイルが未実装

---

### 2.5 入力モダリティ

#### 2.5.1 ポインタのジェスチャ (レベルA)

**説明:**
マルチポイントまたはパスベースのジェスチャで操作できる機能は、パスベースではない単一ポイントで操作できる必要があります(必須の場合を除く)。

**確認方法:**
- ピンチズーム、スワイプ、ドラッグなどのジェスチャに代替操作があるか確認
- シングルタップやクリックで同じ機能が利用できるか確認
- 地図やスライダーが代替方法で操作可能か確認

**よくある失敗例:**
- スワイプのみでカルーセルを操作する
- ピンチズームのみで地図を拡大する
- マルチタッチジェスチャーに代替がない

#### 2.5.2 ポインタのキャンセル (レベルA)

**説明:**
単一ポイント操作で実行される機能は、以下のいずれかを満たす必要があります:
- ダウンイベントで実行されない
- アップイベントで完了し、中止または取り消しが可能
- アップイベントで前のダウンイベントの結果を元に戻す
- ダウンイベントでの完了が必須

**確認方法:**
- ボタンやリンクがマウスボタンを放したとき(アップイベント)に実行されるか確認
- ドラッグ操作をキャンセルできるか確認
- 誤操作を防ぐ仕組みがあるか確認

**よくある失敗例:**
- `mousedown`イベントのみで重要なアクションを実行する
- ドラッグ操作を中断できない
- 誤クリックを取り消す方法がない

#### 2.5.3 名前 (name) のラベル (label) (レベルA)

**説明:**
テキストを含むユーザインターフェースコンポーネントのアクセシブル名には、視覚的に表示されているテキストが含まれている必要があります。

**確認方法:**
- ボタンやリンクの視覚的なテキストとアクセシブル名が一致しているか確認
- `aria-label`が視覚的なラベルと異なっていないか確認
- 音声入力で視覚的なラベルを言って要素を操作できるか確認

**よくある失敗例:**
- ボタンに「送信」と表示されているが`aria-label="Submit form"`のように異なる
- 視覚的なラベル「検索」に対してアクセシブル名が「サイト内検索」
- アイコンボタンに視覚的なテキストがあるのに`aria-label`で上書きしている

#### 2.5.4 動きによる起動 (レベルA)

**説明:**
デバイスの動き(振る、傾けるなど)や方ユーザーの動き(カメラに向かって手を振るなど)で操作できる機能は、ユーザインターフェースコンポーネントでも操作でき、かつ動きへの応答を無効化できる必要があります。

**確認方法:**
- デバイスを振って操作する機能に代替ボタンがあるか確認
- 動きによる操作を無効化する設定があるか確認
- 誤動作を防ぐ仕組みがあるか確認

**よくある失敗例:**
- デバイスを振ることでしか元に戻せない
- 傾けることでしか操作できない機能
- 動きによる操作を無効化できない

---

## 理解可能 (Understandable)

情報とユーザーインターフェースの操作が理解可能である必要があります。

### 3.1 読みやすさ

#### 3.1.1 ページの言語 (レベルA)

**説明:**
各ウェブページのデフォルトの自然言語がプログラムによって解釈可能である必要があります。

**確認方法:**
- `<html>`要素に`lang`属性があるか確認
- `lang`属性の値が正しい言語コード(例: `ja`, `en`)か確認
- スクリーンリーダーが適切な言語で読み上げるか確認

**よくある失敗例:**
- `<html>`に`lang`属性がない
- `lang="en"`なのに日本語コンテンツ
- `lang`の値が無効(例: `lang="japanese"`)

#### 3.1.2 一部分の言語 (レベルAA)

**説明:**
コンテンツの一部が異なる言語で書かれている場合、その部分の言語がプログラムによって解釈可能である必要があります(固有名詞、専門用語などを除く)。

**確認方法:**
- 異なる言語の段落や文に`lang`属性が設定されているか確認
- 引用文や外国語の用語に適切な`lang`属性があるか確認
- スクリーンリーダーが言語を切り替えて読み上げるか確認

**よくある失敗例:**
- 英語の段落に`lang`属性がない日本語ページ
- 複数言語が混在するページで言語指定がない
- 引用文の言語が指定されていない

---

### 3.2 予測可能

#### 3.2.1 フォーカス時 (レベルA)

**説明:**
ユーザインターフェースコンポーネントがフォーカスを受け取ったときに、コンテキストの変化を引き起こしてはいけません。

**確認方法:**
- フォームフィールドにフォーカスしたときに自動送信されないか確認
- タブキーで移動したときにページが変わらないか確認
- フォーカス時にポップアップが開かないか確認

**よくある失敗例:**
- セレクトボックスにフォーカスしただけでページが変わる
- フォーカス時に新しいウィンドウが開く
- フォーカスが自動的に別の要素に移動する

#### 3.2.2 入力時 (レベルA)

**説明:**
設定の変更が自動的にコンテキストの変化を引き起こす場合、ユーザーに事前に知らせる必要があります。

**確認方法:**
- セレクトボックスを変更したときの動作をユーザーが予測できるか確認
- チェックボックスやラジオボタンを変更したときの挙動が明確か確認
- 自動送信される場合、事前に説明があるか確認

**よくある失敗例:**
- セレクトボックスを変更すると警告なしにページが変わる
- ラジオボタンを選択すると自動的にフォームが送信される
- チェックボックスをオンにすると予期しないポップアップが開く

#### 3.2.3 一貫したナビゲーション (レベルAA)

**説明:**
複数のページで繰り返されるナビゲーションの仕組みは、同じ相対的な順序で表示される必要があります(ユーザーが変更した場合を除く)。

**確認方法:**
- ナビゲーションメニューの項目順序が各ページで一貫しているか確認
- ヘッダーやフッターの要素配置が一貫しているか確認
- ユーザーがカスタマイズ可能な場合を除いて順序が変わらないか確認

**よくある失敗例:**
- ページによってメニュー項目の順序が異なる
- 一部のページでナビゲーション項目が欠けている
- モバイルとデスクトップで順序が不必要に異なる

#### 3.2.4 一貫した識別性 (レベルAA)

**説明:**
同じ機能を持つコンポーネントは、一貫して識別される必要があります。

**確認方法:**
- 同じ機能のボタンが同じラベルやアイコンを使用しているか確認
- 「検索」機能が各ページで同じように表現されているか確認
- 同じ意味のアイコンが一貫して使用されているか確認

**よくある失敗例:**
- ページによって「検索」ボタンが「探す」「サーチ」など異なるラベル
- 同じ機能のアイコンが異なる
- 「次へ」ボタンが「次のページ」「続ける」など一貫していない

---

### 3.3 入力支援

#### 3.3.1 エラーの特定 (レベルA)

**説明:**
入力エラーが自動的に検出された場合、エラーの箇所が特定され、テキストでユーザーに説明される必要があります。

**確認方法:**
- フォーム送信時にエラーメッセージが表示されるか確認
- どのフィールドにエラーがあるか明確に示されているか確認
- エラーメッセージがスクリーンリーダーで読み上げられるか確認
- エラーメッセージがフィールドの近くに表示されるか確認

**よくある失敗例:**
- エラーメッセージが色のみで示される
- どのフィールドにエラーがあるか不明
- エラーメッセージが汎用的すぎる(「エラーが発生しました」のみ)
- エラーメッセージがスクリーンリーダーに伝わらない

#### 3.3.2 ラベル又は説明 (レベルA)

**説明:**
ユーザー入力を要求するコンテンツには、ラベルまたは説明が提供される必要があります。

**確認方法:**
- すべてのフォームフィールドにラベルがあるか確認
- ラベルが`<label>`要素またはARIA属性で関連付けられているか確認
- 複雑なフィールドに説明テキストがあるか確認
- プレースホルダーだけでなくラベルがあるか確認

**よくある失敗例:**
- ラベルがなく、プレースホルダーのみ
- ラベルとフィールドが関連付けられていない
- 「メールアドレス」のような説明的なラベルがない(例: 「入力1」)
- ラジオボタンやチェックボックスにラベルがない

#### 3.3.3 エラー修正の 提案 (レベルAA)

**説明:**
入力エラーが自動的に検出され、修正方法が分かる場合、ユーザーに提案を提供する必要があります(セキュリティやコンテンツの目的を損なわない限り)。

**確認方法:**
- エラーメッセージに具体的な修正方法が含まれているか確認
- 「有効なメールアドレスを入力してください」など、どう修正すべきか明確か確認
- 必要に応じて例が示されているか確認

**よくある失敗例:**
- 「入力が無効です」のみで、どう修正すべきか不明
- パスワード要件が明確でない
- 日付フォーマットが説明されていない

#### 3.3.4 エラー回避 (法的、金融、データ) (レベルAA)

**説明:**
法的義務が発生する、金融取引を行う、データを変更/削除するページでは、以下のいずれかが必要です:
- **取消可能:** 送信を取り消せる
- **チェック可能:** データがチェックされ、修正の機会がある
- **確認可能:** 送信前に確認、修正、キャンセルの仕組みがある

**確認方法:**
- 重要な操作に確認ステップがあるか確認
- 送信後に取消や修正ができるか確認
- 入力内容の確認画面があるか確認
- 誤操作を防ぐ仕組みがあるか確認

**よくある失敗例:**
- 購入ボタンをクリックすると即座に決済され、確認がない
- アカウント削除が確認なしで実行される
- 重要なフォームに確認ステップがない
- 送信後に修正や取消ができない

---

## 堅牢 (Robust)

コンテンツは、支援技術を含む様々なユーザーエージェントによって解釈できるように十分に堅牢である必要があります。

### 4.1 互換性

#### 4.1.1 構文解析 (レベルA) 【WCAG 2.2で廃止】

**説明:**
マークアップ言語を使用して実装されたコンテンツでは、要素に完全な開始タグと終了タグがあり、仕様に従ってネストされている必要があります。

**注意:** この基準はWCAG 2.2で廃止されました。現代のブラウザは構文エラーを適切に処理するため、この基準は不要と判断されました。ただし、有効なHTMLを書くことは依然としてベストプラクティスです。

**確認方法:**
- HTML validatorでマークアップを検証
- 開始タグと終了タグが正しくペアになっているか確認
- 要素が適切にネストされているか確認
- 重複するIDがないか確認

**よくある失敗例:**
- 閉じタグがない(`<div><p>テキスト</div>`)
- 不正なネスト(`<p><div>テキスト</div></p>`)
- 重複するID属性
- 無効な属性名や値

#### 4.1.2 名前 (name)、役割 (role)、値 (value) (レベルA)

**説明:**
すべてのユーザインターフェースコンポーネントの名前 (name) と役割 (role) は、プログラムによって解釈可能である必要があります。状態、プロパティ、値もプログラムによって設定可能であり、変更通知が支援技術に提供される必要があります。

**確認方法:**
- 標準的なHTML要素が適切に使用されているか確認
- カスタムウィジェットにARIA role、aria-label、aria-describedbyなどが設定されているか確認
- 動的に変化する状態がaria-expanded、aria-checked、aria-selectedなどで表現されているか確認
- フォームコントロールにラベルが関連付けられているか確認
- スクリーンリーダーで要素の役割、名前、状態が正しく読み上げられるか確認

**よくある失敗例:**
- `<div onclick="...">`のようなボタンを`<button>`や`role="button"`なしで実装
- カスタムチェックボックスに`aria-checked`がない
- アコーディオンに`aria-expanded`がない
- タブに`role="tab"`、`role="tablist"`、`role="tabpanel"`がない
- モーダルに`role="dialog"`、`aria-labelledby`、`aria-modal`がない
- ラベルのない フォームフィールド

#### 4.1.3 ステータスメッセージ (レベルAA)

**説明:**
ステータスメッセージは、スクリーンリーダーなどの支援技術にフォーカスを移動せずに伝えられる必要があります。

**確認方法:**
- 成功メッセージ、エラーメッセージ、進行状況の通知にARIAライブリージョンが使用されているか確認
- `role="status"`, `role="alert"`, `aria-live="polite"`, `aria-live="assertive"`が適切に使用されているか確認
- スクリーンリーダーでメッセージが読み上げられるか確認
- フォーカスが移動しないか確認

**よくある失敗例:**
- 「保存しました」メッセージがARIAライブリージョンなしで表示される
- 検索結果の件数が読み上げられない
- カートに商品を追加しても通知されない
- フォーム送信後の成功メッセージがスクリーンリーダーに伝わらない
- ローディング状態が支援技術に伝わらない

## まとめ

このチェックリストは、WCAG 2.1 レベルA/AAの主要な達成基準をカバーしています。アクセシブルなウェブサイトを構築するためには:

1. **計画段階からアクセシビリティを考慮する**
2. **セマンティックHTMLを使用する**
3. **キーボードとスクリーンリーダーで定期的にテストする**
4. **自動化ツールと手動テストを組み合わせる**
5. **実際のユーザー(障害のある方を含む)からフィードバックを得る**

アクセシビリティは継続的な取り組みであり、一度だけの対応ではありません。新しい機能を追加するたびに、これらの基準を満たしているか確認しましょう。
